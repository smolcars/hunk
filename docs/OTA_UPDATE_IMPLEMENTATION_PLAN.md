# Hunk OTA Update Implementation Plan

## Scope

This document defines the production plan for adding over-the-air updates to Hunk across macOS, Windows, and Linux while keeping packaging stable and simple.

The current product decisions are:

- Hunk will use one release channel only.
- Hunk will stop publishing prerelease versions such as `alpha`, `beta`, `nightly`, or `preview`.
- Hunk release versions will use stable semver only, for example `0.0.1`, `0.1.0`, `0.2.0`.
- Linux package-manager installs such as `.deb` and `.rpm` will not self-update in-app.
- Linux direct bundle installs may self-update.

## Goals

- Let direct-install users update from inside Hunk without manually downloading binaries from GitHub.
- Keep the packaging model understandable and maintainable.
- Avoid introducing a custom release backend when a signed static manifest is sufficient.
- Keep platform-specific update behavior explicit instead of hiding it behind fragile abstractions.

## Non-Goals

- Multiple public release channels.
- In-app self-update for Linux package-manager installs.
- Reworking Hunk packaging formats beyond what is needed for OTA support.

## Current Packaging Baseline

- macOS first-install artifact: signed `.dmg`
- Windows first-install artifact: `.msi`
- Linux direct-install artifact: bundled tarball
- Linux package-manager artifacts: `.deb` and `.rpm`

The OTA design should preserve those first-install artifacts and add updater-specific assets only where needed.

## Recommended Architecture

### 1. New `hunk-updater` crate

Create `crates/hunk-updater` to hold all OTA logic that is not UI-specific:

- updater config
- install source detection
- signed manifest fetch
- version comparison
- update status state machine
- asset download
- signature verification
- platform-specific install orchestration
- relaunch handling

`hunk-desktop` should own GPUI actions, menus, prompts, notifications, and settings wiring. `hunk-domain` should own persisted user config.

### 2. Signed static update manifest

Publish a single manifest for the stable channel, for example:

- `https://pub-de32dfa5fe9845849590fa075f3edafa.r2.dev/stable.json`

Recommended manifest shape:

```json
{
  "version": "0.1.0",
  "pub_date": "2026-04-05T20:00:00Z",
  "notes": "Release notes in markdown or plain text.",
  "platforms": {
    "macos-aarch64": {
      "url": "https://github.com/BlixtWallet/hunk/releases/download/v0.1.0/Hunk.app.tar.gz",
      "signature": "<minisign-or-equivalent-signature>",
      "format": "app"
    },
    "windows-x86_64": {
      "url": "https://github.com/BlixtWallet/hunk/releases/download/v0.1.0/Hunk-0.1.0-x64.msi",
      "signature": "<signature>",
      "format": "wix"
    },
    "linux-x86_64": {
      "url": "https://github.com/BlixtWallet/hunk/releases/download/v0.1.0/Hunk-0.1.0-linux-x86_64.tar.gz",
      "signature": "<signature>",
      "format": "tarball"
    }
  }
}
```

Implementation detail:

- Hunk should own the manifest schema and verifier.
- The shape should stay intentionally close to `cargo-packager-updater` conventions so the design remains familiar and externally inspectable.
- Asset signatures should be generated by a dedicated updater signing key, separate from platform codesigning.

### 3. Install source detection

The updater must detect how Hunk was installed before enabling self-update.

Expected behaviors:

- macOS bundled app install: self-update allowed
- Windows MSI install: self-update allowed
- Linux direct tarball install: self-update allowed
- Linux `.deb` or `.rpm` install: self-update disabled, show explanation

Persisted config should support:

- `auto_update_enabled`
- `last_update_check_at`

Install-source explanation may be derived dynamically from packaging/runtime context, for example via installer-provided environment or launch metadata.

### 4. Updater state model

The updater service should expose explicit states:

- `Idle`
- `Checking`
- `UpdateAvailable`
- `Downloading`
- `Installing`
- `Updated`
- `Errored`
- `DisabledByInstallSource`

That state should drive:

- menu labels
- settings toggle state
- progress notifications
- restart prompts

### 5. GPUI integration

Add the following desktop actions:

- `CheckForUpdates`
- `DismissUpdateError`
- `RestartToApplyUpdate`
- `ViewReleaseNotes`

Surface areas:

- app menu item
- settings pane
- notifications/toasts
- optional About dialog version/update section

## Platform-Specific Design

### macOS

First install remains the signed and notarized `.dmg`.

OTA flow:

1. Download signed `Hunk.app.tar.gz`.
2. Verify updater signature.
3. Extract to a temporary directory.
4. Replace the running `.app` bundle in place.
5. Relaunch Hunk.

Notes:

- Do not use the DMG itself as the OTA payload.
- The OTA tarball should contain the `.app` bundle only.
- Replacement should use a temporary sibling path and atomic rename where possible.
- If the bundle location requires elevation, fail with a clear error instead of inventing a partial fallback path.

### Windows

First install and OTA install both use WiX MSI.

OTA flow:

1. Download signed MSI.
2. Verify updater signature.
3. Invoke `msiexec /i` in passive mode.
4. Exit Hunk.
5. Relaunch Hunk after install completes.

Notes:

- Stable semver only removes the current Windows prerelease version complication.
- Keep MSI as the OTA vehicle instead of introducing a separate installer format.
- Add a helper executable only if normal MSI relaunch/elevation behavior proves unreliable in testing.

### Linux Direct Bundle Install

First install remains the bundled tarball.

OTA flow:

1. Detect the current bundle root from the running executable path.
2. Download signed Linux tarball.
3. Verify updater signature.
4. Extract into a temporary directory on the same filesystem.
5. Replace the existing bundle in place.
6. Relaunch Hunk.

Notes:

- The updater should preserve the current bundle layout used by Hunk packaging.
- Replacement must be same-filesystem to keep rename semantics reliable.
- The updater should not depend on package-manager behavior for this install type.

### Linux Package-Manager Install

Package-managed installs must not self-update.

Behavior:

- disable update install actions
- keep manual check optional if useful
- show a clear message such as:
  `This Hunk install is managed by your Linux package manager. Update it with apt or dnf.`

## Release Pipeline Changes

### Artifact matrix

Release jobs should produce:

- macOS DMG for first install
- macOS `Hunk.app.tar.gz` for OTA
- Windows MSI for first install and OTA
- Linux tarball for first install and OTA
- Linux `.deb`
- Linux `.rpm`

### New publish steps

After artifacts are built:

1. generate SHA256 digests
2. sign OTA assets with updater private key
3. generate `stable.json`
4. upload OTA assets, signatures, and manifest to stable hosting
5. upload install artifacts to GitHub Releases as before

### Versioning rules

- Git tags should use `vX.Y.Z`
- Cargo package version should use `X.Y.Z`
- No prerelease suffixes
- No channel-specific artifact names

## Incremental Execution Phases

### Phase 1: Foundations

- normalize versioning to stable semver only
- remove alpha/prerelease release assumptions
- add this design document
- add updater config fields in `hunk-domain`
- add `crates/hunk-updater` skeleton

### Phase 2: Release Pipeline

- generate OTA-safe macOS tarball
- sign OTA assets
- generate and publish `stable.json`
- document updater key management

### Phase 3: Desktop Integration

- add updater actions to `hunk-desktop`
- add menu item and notification flow
- add settings controls
- add restart prompt behavior

### Phase 4: macOS Updater

- manifest fetch
- signature verification
- tarball extraction
- app bundle replacement
- relaunch

### Phase 5: Windows Updater

- manifest fetch
- signature verification
- MSI install flow
- relaunch
- elevation/restart validation

### Phase 6: Linux Updater

- direct-install detection
- tarball install flow
- package-manager detection and explanation messaging
- relaunch

### Phase 7: Verification

- unit tests for versioning, manifest parsing, and signature verification
- integration tests for install-source detection
- smoke tests for each OTA-supported platform
- packaging validation after updater asset generation

## Risk Areas

### 1. Windows installer version monotonicity

MSI upgrades require predictable version progression. Stable semver only is the simplest answer and is now the required policy.

### 2. macOS replacement correctness

Replacing an app bundle in place is safe only if extraction, staging, and rename behavior are explicit and fully tested.

### 3. Linux install source ambiguity

The updater must not guess incorrectly and mutate package-managed paths.

### 4. Signature verification

OTA verification is not optional. Platform codesigning alone is not enough for the updater trust path.
